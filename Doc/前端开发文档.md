## 2. 设计系统（Design System）

### 2.1 设计哲学

**Wikipedia/Notion 文学风**：画布是安静的"书页"，节点是"段落卡片"，操作是低调的。

| 维度 | 取向 | 说明 |
|------|------|------|
| 字体 | 节点内容用衬线，UI 控件用无衬线 | 内容：Georgia/Noto Serif SC；UI：Inter/system-ui |
| 色调 | Notion 暖白底 | 避免纯白 `#FFFFFF`，采用暖白 `#FAFAF8` |
| 信息密度 | Notion 块状布局 | 节点卡片 = block，留白充足 |
| 交互感 | Notion 悬浮反馈 | 悬浮高亮 + 微阴影 + 微动效，操作低调 |

## 4. 页面布局与交互状态

### 4.1 三窗格布局（Tri-pane）


### 4.2 路由

MVP 阶段只有一个主路由：**GraphWorkspace**。

| 路由 | 组件 | 说明 |
|------|------|------|
| `/` | `GraphWorkspace` | 三窗格主页面，通过 UI 状态切换交互模式 |

后续可扩展：`/audit` 审计日志全览、`/login` 登录等。

### 4.3 GraphWorkspace 四种交互状态

四页设计图对应同一 `GraphWorkspace` 组件的 4 种交互状态，由 `uiStore.rightPanelMode` 驱动：

| 状态 | 设计图 | 左栏 | 画布 | 右面板 | 触发条件 |
|------|-------|------|------|--------|---------|
| **默认浏览** | 页 1 | RhizoneList 可见 | 全量 DAG 渲染 | 隐藏 | 初始加载 / 取消选中 |
| **节点选中** | 页 2 | 保持 | 居中选中节点 + 高亮邻居 | NodeDetailPanel（含 Tabs） | 点击任意节点 |
| **△ 操作** | 页 3 | 保持 | 高亮源节点 + 新节点占位 | EditDraftPanel（InjectForm / ForkForm） | 点击 NodeActionToolbar 菜单项 |
| **Agent 模式** | 页 4 | 保持 | Agent 产出节点动画 | AgentPanel（v0.3 暂缓） | Agent 决策执行中 |

## 7. 组件设计规范（35 项）

### 7.1 图谱画布组件

#### #1–#3 节点卡片（HumanPostNode / ConsensusNode / ResultNode）

**形态变化**：节点随画布缩放级别在圆形与方形之间过渡。

| 缩放级别 | 形态 | 尺寸 | 内容显示 |
|---------|------|------|---------|
| < 0.5x（远景） | 圆形 | 40x40px | 仅边框颜色区分类型，无文字 |
| 0.5x–0.8x（中景） | 圆形 | 60x60px | 单行标题，溢出省略号 |
| > 0.8x（近景） | 圆角矩形 | 自适应宽度（max 320px），高度随内容 | 完整卡片布局 |

**圆形 → 方形过渡动画**：`border-radius` 从 `9999px` 过渡到 `8px`，`width/height` 从固定值过渡到自适应，时长 `--transition-normal`（200ms）。

**近景卡片结构**：
```
┌─ 左边框 2px 类型色 ──────────────────┐
│ [类型图标] 标题/作者          时间     │  ← font-ui, font-size-xs
│                                      │
│ 正文内容（最多 3 行省略）              │  ← font-content, font-size-base
│                                      │
│ [标签]                               │  ← 可选：embedding 状态等
└──────────────────────────────────────┘
```

**类型区分**（仅通过左侧边框颜色）：

| 类型 | 边框色 |
|------|--------|
| Human_Post | `--color-node-human`（#2E7CF6） |
| AI_Consensus | `--color-node-consensus`（#9B59B6） |
| Result | `--color-node-result`（#0D9488） |

**交互状态**：

### 8.2 自定义边注册

语义关联边（`CONCEPTUAL_OVERLAP`、`RELATES_TO`）不在 DAG 画布主视图显示，仅在 AssociationPanel 中呈现。

### 8.3 关系感知 Track 布局

主画布不再使用“所有关系同一规则”的通用 Dagre 直接布局，而是使用**版本关系感知**的 Track 布局。目标是同时满足：

- `CONTINUES_FROM`：像 Git 主线，沿当前讨论车道向下延续
- `BRANCHED_FROM`：像 Neo4j Graph 的局部展开，从源节点侧向拉出平行异议分支
- `MERGED_INTO` / `CONVERGED_FROM`：保持向上收敛，不与主线/分支语义混淆
- 语义关联边（`CONCEPTUAL_OVERLAP`、`RELATES_TO`）不进入主画布布局计算

**布局原则：**

1. 每个 root 图谱有一个中心主车道（lane = 0）。
2. `CONTINUES_FROM` 继承父节点车道，`rank + 1` 向下排列。
3. `BRANCHED_FROM` 与源节点保持同一 `rank`，但分配到侧向 branch lane。
4. 同一源节点的多个分支使用稳定槽位：`+1, -1, +2, -2 ...`，排序键为 `created_at` + `node_id`。
5. 分支节点自己的延续继续沿该 branch lane 向下，不回跳到主车道。
6. 主视图只表达版本演进层，不渲染语义关联层。

**示意：**

```text
      [B2]
        \
[B1] --- [R]
          |
         [C1]
          |
         [C2]
```

- `R`：当前源节点
- `B1/B2`：`BRANCHED_FROM`，侧向平行分支
- `C1/C2`：`CONTINUES_FROM`，纵向主线延续

**句柄规则：**

- `CONTINUES_FROM`：`source-top -> target-bottom`
- `BRANCHED_FROM`：左侧分支走 `source-right -> target-left`，右侧分支走 `source-left -> target-right`
- `MERGED_INTO` / `CONVERGED_FROM`：保持纵向句柄，但在视觉样式上弱于主线

**边路由规则：**

- `CONTINUES_FROM`：优先竖向正交线，形成清晰主线
- `BRANCHED_FROM`：优先侧向肘形线或轻曲线，避免 root 多分支共用同一个底部出口
- 不允许所有版本边复用同一条 `smoothstep` 路径策略

### 8.3.1 双模式画布

在保留 `Track` 主视图的前提下，前端新增 `canvasMode`：

- `lineage`：默认模式。继续使用关系感知 Track 布局，负责稳定表达版本演进。
- `explore`：探索模式。使用 `React Flow + d3-force` 对当前 root 局部子图做物理模拟，强调 Neo4j 风格的局部关系探索。

**模式职责：**

1. `lineage` 是主真相视图，强调可追溯、稳定、方向明确的版本 DAG。
2. `explore` 是局部探索视图，强调拖拽、回弹、避让与邻接关系观察。
3. 两种模式共用同一份后端 `nodes[] + edges[]` 语义数据，只是布局与交互不同。

**探索模式规则：**

1. 当前选中节点优先作为物理锚点；若没有选中节点，则使用当前 root。
2. `CONTINUES_FROM` 给予更强纵向偏置，保留“上下游演进”感。
3. `BRANCHED_FROM` 给予横向展开偏置，同一源节点的多个分支按稳定顺序分布到左右两侧。
4. `MERGED_INTO` / `CONVERGED_FROM` 给予向上收敛偏置，但边样式弱于主线。
5. 节点可自由拖拽；拖拽时临时固定，释放后恢复到力导平衡。

```typescript
const NODE_TYPE_MAP: Record<NodeLabel, string> = {

}
```

## 9. SSE 实时对接

### 9.1 连接管理（fetch + ReadableStream）

后端 `JwtAuthenticationFilter` 仅从 `Authorization` header 读取 token，原生 `EventSource` 不支持自定义 header。因此采用 `fetch + ReadableStream` 方案：

```typescript
// hooks/useSse.ts
const SSE_URL = "/api/events/stream";

async function connectSse(token: string, graphStore: GraphState) {
  const res = await fetch(SSE_URL, {
    headers: { Authorization: `Bearer ${token}` },
  });
  if (!res.ok || !res.body) throw new Error("SSE connection failed");

  const reader = res.body.pipeThrough(new TextDecoderStream()).getReader();
  let buffer = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += value;

    // 按 SSE 协议解析 event: / data: 行
    const lines = buffer.split("\n\n");
    buffer = lines.pop() ?? "";
    for (const block of lines) {
      const event = parseSseBlock(block);
      if (!event) continue;
      handleSseEvent(event, graphStore);
    }
  }
}

function handleSseEvent(
  event: { type: string; data: string },
  graphStore: GraphState
) {
  switch (event.type) {
    case "NODE_CREATED": {
      const payload: NodeCreatedEvent = JSON.parse(event.data);
      fetchNode(payload.node_id).then((node) => graphStore.addNode(node));
      break;
    }
    case "EDGE_CREATED": {
      const payload: EdgeCreatedEvent = JSON.parse(event.data);
      graphStore.addEdge(payload);
      break;
    }
    case "DECISION_COMPLETE": {
      const payload: DecisionCompleteEvent = JSON.parse(event.data);
      switch (payload.decision_type) {
        case "MERGE":
        case "JOIN":
          // 共识类：刷新 lineage 以获取汇聚结果
          break;
        case "BRANCH":
        case "INJECT":
        case "FORK":
          // 分支类：可高亮新分支节点
          break;
        case "MATERIALIZE":
        case "CROSS_SYNTH":
          // 结果类：刷新关联的 Result 节点
          break;
      }
      break;
    }
  }
}
```

### 9.2 增量更新策略

收到 SSE 事件后不直接重新布局整个图，而是：

1. `NODE_CREATED` → 拉取节点详情 → 加入 `nodes` Map → 追加到 `rfNodes` → 仅对新节点计算位置
2. `EDGE_CREATED` → 追加到 `edges` + `rfEdges`
3. **批量更新**：500ms 内的多个事件合并为一次 React Flow 更新，避免高频重渲染

### 10.1 初始加载

### 10.9 决策类型与画布响应

后端 7 种决策执行后，前端通过 SSE 接收增量更新。各决策对画布的影响：

| 决策类型 | 产出节点 | 产出边 | 车道规则 | 句柄/边规则 | 画布行为 |
|---------|---------|--------|---------|-------------|---------|
| MERGE | AI_Consensus | MERGED_INTO + SYNTHESIZED_FROM | 收敛到上游共识层 | 纵向句柄，上收敛边 | 在源节点上方插入共识节点 |
| BRANCH | Human_Post | BRANCHED_FROM | 与源节点同 rank，分配侧向 branch lane | 左右句柄 + 侧向肘形线 | 在源节点旁拉出平行分支 |
| INJECT | Human_Post | CONTINUES_FROM | 继承父节点 lane，`rank + 1` | `source-top -> target-bottom` 竖向主线 | 在源节点下方追加延续节点 |
| MATERIALIZE | Result | MATERIALIZED_FROM | 侧向附着，不参与主线 lane 继承 | 侧向辅助边 | 在源节点旁生成绿色结果节点 |
| FORK | N 个 Human_Post | N 个 BRANCHED_FROM | 同源节点按 `+1, -1, +2, -2` 分配稳定分支槽位 | 左右句柄 + 多条独立侧向边 | 在源节点旁扇形展开多个分支 |
| CROSS_SYNTH | Result | CROSS_SYNTHESIZED_FROM | 独立结果层 | 弱化连接，避免干扰版本主视图 | 连接多个 Result 生成新 Result |
| JOIN | AI_Consensus | CONVERGED_FROM | 多源向同一共识节点收束 | 纵向汇聚边 | 多个源节点汇聚为一个共识节点 |

**补充约束：**

- `BRANCHED_FROM` 必须被视为“平行异议分支”，不能降格成向下延续。
- `CONTINUES_FROM` 必须保持 Git 风格主线视觉稳定性。
- 同一源节点的多分支在刷新前后必须保持左右顺序稳定，不能因重排抖动。
- `explore` 模式不得改写版本关系语义，只允许改写节点坐标与边路径表现。
- SSE 增量节点/边到达后，`lineage` 与 `explore` 都必须基于同一语义数据刷新，不能出现“一个模式有数据、另一个模式缺失”的分裂状态。

```typescript
// api/client.ts
const BASE_URL = import.meta.env.VITE_API_BASE_URL ?? "";

async function request<T>(path: string, options?: RequestInit): Promise<T> {
  const token = localStorage.getItem("jwt_token");
  const res = await fetch(`${BASE_URL}${path}`, {
    ...options,
    headers: {
      "Content-Type": "application/json",
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options?.headers,
    },
  });
  const body: ApiResponse<T> = await res.json();
  if (body.code !== 0) {
    throw new Error(body.message);
  }
  return body.data;
}
```

```typescript
// authStore.ts 集成示意
const useAuthStore = create<AuthState>((set, get) => ({
  token: localStorage.getItem("jwt_token"),
  roles: [],

  setToken: (token: string) => {
    localStorage.setItem("jwt_token", token);
    // 解析 JWT payload 获取 roles
    const payload = JSON.parse(atob(token.split(".")[1]));
    set({ token, roles: payload.roles ?? [] });
  },

  clearToken: () => {
    localStorage.removeItem("jwt_token");
    set({ token: null, roles: [] });
  },

  isAuthenticated: () => get().token !== null,
  hasRole: (role: string) => get().roles.includes(role),
}));
```
