# RhizoDelta 使用手册

本文档面向当前仓库的开发者和本地调试人员，目标是回答三个问题：

1. 项目怎么跑起来
2. API 怎么调
3. 前端页面怎么用

本文内容基于当前仓库里的真实实现整理而成，不复述白皮书式设计说明。

## 1. 项目组成

当前仓库由四部分组成：

- Spring Boot 后端
- React + Vite 前端
- Neo4j 图数据库
- RabbitMQ / Redis 中间件

关键入口：

- 后端启动类：`src/main/java/com/rhizodelta/RhizoDeltaApplication.java`
- 前端入口：`frontend/src/App.tsx`
- 主工作区：`frontend/src/components/GraphWorkspace.tsx`
- Docker 编排：`docker-compose.yml`

## 2. 运行前准备

本地运行至少需要：

- JDK 17+
- Maven
- Node.js 与 npm
- Docker Compose
- 可用的 `SILICON_FLOW_API_KEY`

注意：

- 后端启动时会校验 LLM 配置中的 API Key
- 如果 `SILICON_FLOW_API_KEY` 为空，应用会在启动阶段直接失败
- Neo4j 在启动时会做连通性校验，不通也会直接失败

## 3. 配置文件说明

### 3.1 `.env`

`.env` 主要给 `docker compose` 使用。

初始化方式：

```bash
cp .env.example .env
```

当前 `docker-compose.yml` 会读取：

- Neo4j 容器名、镜像版本、端口、认证信息
- RabbitMQ / Redis 端口和账户，部分字段带默认值

### 3.2 `application.yml`

`src/main/resources/application.yml` 是通用配置入口，里面定义了：

- 默认 profile：`${SPRING_PROFILES_ACTIVE:local}`
- Neo4j 环境变量读取方式
- 向量维度、AI 召回和预算参数
- JWT secret
- LangChain4j 的 chat / embedding 模型配置

### 3.3 `application-local.yml`

本地开发最关键的是 `src/main/resources/application-local.yml`。
它当前直接写死了本地依赖地址：

- Neo4j：`bolt://localhost:7687`
- RabbitMQ：`localhost:5672`
- Redis：`localhost:6379`

同时它还定义了本地 JWT secret，并通过环境变量读取：

- `SILICON_FLOW_API_KEY`

### 3.4 一个很容易踩的坑

`.env` 中的 Neo4j 密码和 `application-local.yml` 里的密码必须一致。

当前 `application-local.yml` 里写的是：

- 用户名：`neo4j`
- 密码：`12345678`

因此建议把 `.env` 也改成一致的值：

```dotenv
NEO4J_AUTH=neo4j/12345678
NEO4J_USERNAME=neo4j
NEO4J_PASSWORD=12345678
NEO4J_URI=bolt://localhost:7687
```

如果你想换密码，需要同步修改两边。

## 4. 启动本地依赖

启动依赖服务：

```bash
docker compose up -d neo4j rabbitmq redis
```

常用访问地址：

- Neo4j Browser: `http://localhost:7474`
- Neo4j Bolt: `bolt://localhost:7687`
- RabbitMQ Management: `http://localhost:15672`
- Redis: `localhost:6379`

检查容器状态：

```bash
docker compose ps
```

## 5. 启动后端

### 5.1 导出 API Key

后端启动前，先导出：

```bash
export SILICON_FLOW_API_KEY=your_api_key_here
```

### 5.2 启动命令

在仓库根目录执行：

```bash
mvn spring-boot:run
```

因为默认 profile 是 `local`，通常不需要额外指定
`SPRING_PROFILES_ACTIVE`。

### 5.3 启动时会发生什么

后端启动过程中，当前代码会做这些检查：

1. 校验 Neo4j 连通性
2. 自动创建约束、普通索引和向量索引
3. 校验 chat model 和 embedding model 的 API Key 不是空值、也不是占位符

如果启动失败，优先检查：

- Neo4j 是否真的已经起来
- Neo4j 密码是否和 `application-local.yml` 一致
- `SILICON_FLOW_API_KEY` 是否已导出

## 6. 启动前端

进入前端目录：

```bash
cd frontend
```

安装依赖：

```bash
npm install
```

启动开发服务器：

```bash
npm run dev
```

### 6.1 API 地址规则

前端请求封装在 `frontend/src/api/client.ts`。

规则如下：

- 如果没有设置 `VITE_API_BASE_URL`，则请求相对路径 `/api/...`
- 开发模式下，Vite 会把 `/api` 代理到 `http://localhost:8080`

因此最常见的本地组合是：

- 前端：`http://localhost:5173`
- 后端：`http://localhost:8080`

此时不需要额外设置 `VITE_API_BASE_URL`。

如果你把后端起在别的地址，可以这样启动前端：

```bash
VITE_API_BASE_URL=http://localhost:8080 npm run dev
```

## 7. JWT 本地调试

### 7.1 当前项目没有登录接口

这一点必须先明确：

- 当前仓库没有登录页
- 当前仓库没有签发 JWT 的 HTTP 接口
- 前端认证完全依赖浏览器里的 `localStorage.jwt_token`

相关实现位置：

- `frontend/src/stores/authStore.ts`
- `frontend/src/api/client.ts`
- `frontend/src/hooks/useSse.ts`

### 7.2 Token 需要包含什么

后端会读取：

- `sub`
- `roles`

前端也会直接解析 `roles` 来显示角色徽标和决定交互能力。

推荐 payload：

```json
{
  "sub": "dev-admin",
  "roles": ["ADMIN"],
  "iat": 1710000000,
  "exp": 1710003600
}
```

注意：

- 角色值写 `ADMIN` / `AGENT` / `USER`
- 不要写 `ROLE_ADMIN`
- 后端会在内部映射成 `ROLE_` 前缀权限

### 7.3 生成 JWT

把下面命令中的 `JWT_SECRET` 替换成
`src/main/resources/application-local.yml` 中
`rhizodelta.jwt.secret` 的当前值：

```bash
JWT_SECRET='replace_with_local_jwt_secret' node - <<'EOF'
const crypto = require("crypto");

const secret = process.env.JWT_SECRET;
if (!secret) {
  throw new Error("JWT_SECRET is required");
}

const now = Math.floor(Date.now() / 1000);
const header = { alg: "HS256", typ: "JWT" };
const payload = {
  sub: "dev-admin",
  roles: ["ADMIN"],
  iat: now,
  exp: now + 3600,
};

const encode = (value) =>
  Buffer.from(JSON.stringify(value)).toString("base64url");

const unsigned = `${encode(header)}.${encode(payload)}`;
const signature = crypto
  .createHmac("sha256", secret)
  .update(unsigned)
  .digest("base64url");

console.log(`${unsigned}.${signature}`);
EOF
```

### 7.4 写入浏览器

打开前端页面后，在浏览器控制台执行：

```js
localStorage.setItem("jwt_token", "paste_your_token_here");
location.reload();
```

如果只是想切角色，可以只改 payload 里的 `roles`：

- `["USER"]`
- `["AGENT"]`
- `["ADMIN"]`

## 8. 权限矩阵

当前权限规则由 `SecurityConfig` 决定，可概括为：

### 8.1 任何已认证用户

只要带有效 Bearer Token，就可以访问：

- `POST /api/posts`
- `GET /api/events/stream`
- `GET /api/nodes/**`
- `GET /api/audit/decisions/**`

### 8.2 `AGENT` 或 `ADMIN`

可以访问：

- `POST /api/decisions/**`
- `POST /api/associations`
- `PUT /api/nodes/{id}/embedding`
- `GET /api/reviews/**`

### 8.3 仅 `ADMIN`

可以访问：

- `POST /api/reviews/**`
- `DELETE /api/associations/{associationId}`
- `POST /api/decisions/{decision_id}/rollback`
- `POST /api/decisions/fork/{operation_id}/rollback`

## 9. 核心 API 用法

所有接口统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

下面的示例都默认你已经准备好了：

```bash
export TOKEN='paste_your_jwt_here'
```

### 9.1 发布新话题

```bash
curl -X POST http://localhost:8080/api/posts \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "request_id": "req-1001",
    "author_id": "author-001",
    "content": "hello graph"
  }'
```

说明：

- 返回 `202 Accepted`
- `event_id` 是根据 `request_id` 生成的稳定 UUID
- 请求不会同步写图，而是先进入 RabbitMQ

### 9.2 回复某个节点

和发布新话题一样，只是多带一个 `target_node_id`：

```bash
curl -X POST http://localhost:8080/api/posts \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "request_id": "req-1002",
    "author_id": "author-001",
    "content": "this is a reply",
    "target_node_id": "replace-with-node-uuid"
  }'
```

### 9.3 查询根话题

```bash
curl http://localhost:8080/api/nodes/roots?limit=20 \
  -H "Authorization: Bearer $TOKEN"
```

### 9.4 查询节点详情

```bash
curl http://localhost:8080/api/nodes/{node_id} \
  -H "Authorization: Bearer $TOKEN"
```

### 9.5 查询 lineage

```bash
curl "http://localhost:8080/api/nodes/{node_id}/lineage?max_depth=3" \
  -H "Authorization: Bearer $TOKEN"
```

### 9.6 查询 children

```bash
curl "http://localhost:8080/api/nodes/{node_id}/children?max_depth=2&limit=50" \
  -H "Authorization: Bearer $TOKEN"
```

### 9.7 查询 provenance

```bash
curl http://localhost:8080/api/nodes/{node_id}/provenance \
  -H "Authorization: Bearer $TOKEN"
```

### 9.8 创建延续注入决策

需要 `AGENT` 或 `ADMIN`：

```bash
curl -X POST http://localhost:8080/api/decisions/inject \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "decision_id": "8cbd16d4-6834-4622-9418-6628149a5e0f",
    "request_id": "4d11d5af-52a4-4971-a265-dc8cc61145bf",
    "source_node_id": "replace-with-node-uuid",
    "content": "继续沿着这个节点补充论证",
    "author_id": "dev-admin",
    "operator_type": "HUMAN",
    "operator_id": "dev-admin",
    "reason": "补充延续内容"
  }'
```

### 9.9 创建分叉决策

需要 `AGENT` 或 `ADMIN`：

```bash
curl -X POST http://localhost:8080/api/decisions/fork \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "operation_id": "5d221b87-ef13-4137-a1f3-f78fa3ed0d59",
    "request_id": "1a8df4bb-4412-46db-9134-d76a6931148d",
    "source_node_id": "replace-with-node-uuid",
    "branches": [
      {
        "decision_id": "17759397-e204-441c-aec9-f0db07151115",
        "content": "第一条分支",
        "author_id": "dev-admin"
      },
      {
        "decision_id": "93ad6825-9803-4c15-a19f-1f9555930f82",
        "content": "第二条分支",
        "author_id": "dev-admin"
      }
    ],
    "operator_type": "HUMAN",
    "operator_id": "dev-admin",
    "reason": "拆成两个平行观点"
  }'
```

### 9.10 创建语义关联

需要 `AGENT` 或 `ADMIN`：

```bash
curl -X POST http://localhost:8080/api/associations \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "source_node_id": "replace-with-source-node-uuid",
    "target_node_id": "replace-with-target-node-uuid",
    "type": "RELATES_TO",
    "creator_id": "dev-admin",
    "reason": "两者在主题上有关联",
    "confidence": 0.82
  }'
```

### 9.11 查询审计记录

```bash
curl "http://localhost:8080/api/audit/decisions?limit=20" \
  -H "Authorization: Bearer $TOKEN"
```

### 9.12 建立 SSE 连接

```bash
curl -N http://localhost:8080/api/events/stream \
  -H "Authorization: Bearer $TOKEN"
```

你会看到类似这些事件名：

- `ORCHESTRATION_STATUS`
- `NODE_CREATED`
- `EDGE_CREATED`
- `DECISION_COMPLETE`

## 10. 前端页面操作

主界面位于 `frontend/src/components/GraphWorkspace.tsx`。

### 10.1 左侧栏

左侧栏组件是 `RhizoneList`，主要功能：

- 展示根话题列表
- 点击根话题后加载对应图谱
- 通过底部按钮发起新话题

### 10.2 画布模式

画布上方有模式切换：

- `版本视图`
- `探索视图`

版本视图更偏 DAG lineage，探索视图更偏自由浏览。

### 10.3 节点详情

点击节点后，可打开右侧详情面板。
当前面板包含 4 个标签页：

- `详情`
- `确权溯源`
- `关联`
- `审计`

### 10.4 节点操作

节点工具条当前支持：

- `延续注入`
- `分叉`

它们会打开右侧编辑面板，并调用：

- `POST /api/decisions/inject`
- `POST /api/decisions/fork`

### 10.5 发布新话题

左下角 `+ 发起新话题` 会打开 `PostForm`。

如果当前已经选中了某个节点，表单会自动把它作为
`target_node_id`，也就是“回复该节点”。

## 11. 异步编排与实时更新

发帖后并不会同步直接返回最终节点。
当前流程是：

1. `POST /api/posts`
2. 后端校验请求
3. 消息写入 RabbitMQ
4. 立即返回 `202`
5. SSE 推送 `ORCHESTRATION_STATUS`
6. 后续根据编排结果继续推送：
   - `NODE_CREATED`
   - `EDGE_CREATED`
   - `DECISION_COMPLETE`

这也是为什么页面右上角的 SSE 状态是重要观察点。

## 12. 常见问题排查

### 12.1 后端一启动就失败

优先排查：

- `SILICON_FLOW_API_KEY` 没有导出
- 还在使用占位符 API Key
- Neo4j 没起来
- Neo4j 用户名或密码不匹配

### 12.2 如果启动失败并提示 UserAccount user_id integrity violation

这说明启动阶段的只读预检发现了 `UserAccount.user_id` 数据不满足约束，
应用会在创建 `rhizodelta_user_account_user_id_unique` 之前直接中止。

常见报错样式：

- `UserAccount user_id integrity violation: blank user_id usernames=[...]`
- `UserAccount user_id integrity violation: duplicate user_id entries=[...]`

#### 场景 A：存在 `null` 或空白 `user_id`

先查出问题行：

```cypher
MATCH (u:UserAccount)
WHERE u.user_id IS NULL OR trim(u.user_id) = ''
RETURN u.username AS username, u.user_id AS user_id
ORDER BY username
LIMIT 20;
```

如果确认这些账号是需要保留的测试/开发数据，可以直接补一个新 UUID：

```cypher
MATCH (u:UserAccount)
WHERE u.user_id IS NULL OR trim(u.user_id) = ''
SET u.user_id = randomUUID()
RETURN u.username AS username, u.user_id AS repaired_user_id
ORDER BY username;
```

执行后重启应用。

注意：

- 这条修复会改变账号的 canonical identity，只适合本地或测试数据
- 如果这些账号已经被外部系统引用，不要直接改 UUID，应先核对来源

#### 场景 B：存在重复 `user_id`

先查出重复值和对应用户名：

```cypher
MATCH (u:UserAccount)
WITH u.user_id AS uid, collect(u.username) AS usernames, count(*) AS n
WHERE n > 1
RETURN uid, usernames, n
ORDER BY uid
LIMIT 20;
```

如果确认其中某条是无效测试数据，可以删除冲突行：

```cypher
MATCH (u:UserAccount {username: $username})
DETACH DELETE u;
```

例如在 Neo4j Browser 里把 `$username` 替换成实际用户名后执行。

执行删除后，再重新运行上面的重复检查查询，确认该 `uid` 只剩一条记录，
然后再重启应用。

注意：

- 不要一次性删除整组重复记录，至少保留一条权威数据
- 生产环境如果无法判断哪条该删，先导出结果并交给业务或 DBA 确认

### 12.3 前端所有请求都返回 401

优先排查：

- 浏览器里是否存在 `localStorage.jwt_token`
- token 是否过期
- token 的签名 secret 是否和后端当前使用的 JWT secret 一致

### 12.4 可以打开前端，但图不刷新

优先排查：

- 页面右上角 SSE 状态是否已断开
- `/api/events/stream` 是否返回 401
- 后端是否连上 RabbitMQ

### 12.5 发帖返回 503

这通常说明消息代理不可用。
当前发帖接口在 RabbitMQ 不可用时会返回：

- HTTP `503`
- 消息：`message broker unavailable`

### 12.6 切到前端后看不到任何话题

优先排查：

- Neo4j 里是否已有根节点
- 是否已经成功发过至少一个帖子
- 当前 token 是否有效，`GET /api/nodes/roots` 是否已被授权
